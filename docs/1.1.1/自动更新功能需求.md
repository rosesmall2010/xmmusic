## 🎯 需求概述

### 背景

当前 XMMusic 应用需要通过手动下载安装包来更新，用户体验不够友好。参考 VSCode 的自动更新机制，我们需要实现一套完整的自动更新系统，让用户能够自动检测、下载和安装新版本。

### 目标

实现类似 VSCode 的自动更新功能，包括：
- 自动检测新版本
- 后台下载更新包
- 用户友好的更新提示
- 自动安装和重启
- 支持手动检查更新

### 价值

- **提升用户体验**: 用户无需手动下载和安装更新
- **提高更新率**: 自动更新机制能提高用户更新到最新版本的比率
- **降低支持成本**: 减少用户因版本问题产生的咨询
- **快速修复问题**: 可以快速推送 bug 修复和安全更新

## 📝 功能需求

### FR1: 版本检测功能

**FR1.1 自动检测更新**
- 应用启动时自动检查是否有新版本（可选，可通过设置控制）
- 定期检查更新（默认每 24 小时检查一次，可通过设置调整）
- 检查更新时不应阻塞主界面

**FR1.2 手动检查更新**
- 在设置页面提供"检查更新"按钮
- 点击后立即检查是否有新版本
- 显示检查进度和结果

**FR1.3 版本信息获取**
- 从 GitHub Releases API 获取最新版本信息
- 比较当前版本与最新版本
- 支持语义化版本号比较（如 1.1.0 vs 1.1.1）

### FR2: 更新下载功能

**FR2.1 后台下载**
- 检测到新版本后，在后台自动下载更新包
- 下载过程中不阻塞用户正常使用应用
- 显示下载进度（百分比、速度、剩余时间）

**FR2.2 下载管理**
- 支持断点续传（下载中断后可以继续）
- 下载失败时自动重试（最多 3 次）
- 下载完成后验证文件完整性（校验和）

**FR2.3 平台支持**
- macOS: 支持 x64 和 arm64 架构的 .dmg 或 .zip 文件
- Windows: 支持 x64 架构的 .exe 安装包
- Linux: 支持 AppImage、deb、rpm 格式（可选）

### FR3: 更新提示功能

**FR3.1 更新通知**
- 检测到新版本时显示通知（系统通知或应用内通知）
- 通知包含版本号、更新内容摘要
- 提供"立即更新"、"稍后提醒"、"忽略此版本"选项

**FR3.2 更新对话框**
- 显示详细的更新信息（版本号、发布日期、更新内容）
- 显示下载进度（下载中）
- 提供"立即安装"、"稍后安装"、"跳过此版本"选项

**FR3.3 更新内容展示**
- 从 GitHub Release 的 Release Notes 中提取更新内容
- 支持 Markdown 格式的更新说明
- 显示主要更新、修复、改进等分类信息

### FR4: 更新安装功能

**FR4.1 自动安装**
- 下载完成后，提示用户是否立即安装
- 用户确认后，退出应用并启动安装程序
- 安装完成后自动启动新版本应用

**FR4.2 延迟安装**
- 用户可以选择"稍后安装"
- 在应用退出时提示是否安装更新
- 支持设置"自动安装更新"选项（无需用户确认）

**FR4.3 安装流程**
- macOS: 挂载 .dmg 文件，复制应用到 Applications 目录
- Windows: 运行 .exe 安装程序，替换旧版本
- 保留用户数据和配置

### FR5: 设置管理功能

**FR5.1 更新设置**
- 启用/禁用自动检查更新
- 设置检查更新频率（每小时、每天、每周、手动）
- 启用/禁用自动下载更新
- 启用/禁用自动安装更新

**FR5.2 更新历史**
- 记录更新历史（版本号、更新时间）
- 显示当前版本信息
- 显示上次检查更新时间

## 🔧 非功能需求

### NFR1: 性能要求

- 版本检查请求应在 3 秒内完成
- 下载更新不应影响应用正常使用（限制下载速度或使用低优先级）
- 更新检查失败不应影响应用启动和运行

### NFR2: 可靠性要求

- 网络错误时优雅处理，不显示错误弹窗
- 下载失败时提供重试机制
- 更新过程中应用崩溃时，下次启动能继续更新流程

### NFR3: 安全性要求

- 验证更新包的签名和完整性
- 使用 HTTPS 下载更新包
- 防止中间人攻击和恶意更新

### NFR4: 用户体验要求

- 更新过程对用户透明，不干扰正常使用
- 提供清晰的进度反馈
- 支持取消下载和安装
- 错误信息清晰易懂

### NFR5: 兼容性要求

- 支持 macOS 10.13+（x64 和 arm64）
- 支持 Windows 10+
- 支持 Linux（Ubuntu 18.04+, Debian 10+, Fedora 30+）
- 向后兼容旧版本的用户数据

## 👥 用户场景

### 场景 1: 自动检测并更新

**用户**: 普通用户
**目标**: 自动获得最新版本，无需手动操作

**流程**:
1. 用户正常使用应用
2. 应用在后台检测到新版本（1.1.2）
3. 显示系统通知："发现新版本 1.1.2，点击查看详情"
4. 用户点击通知，打开更新对话框
5. 用户点击"立即更新"
6. 应用在后台下载更新包，显示进度
7. 下载完成后，提示"更新已就绪，是否立即安装？"
8. 用户点击"立即安装"
9. 应用退出，安装更新，自动启动新版本

### 场景 2: 手动检查更新

**用户**: 技术用户
**目标**: 主动检查是否有更新

**流程**:
1. 用户打开设置页面
2. 点击"检查更新"按钮
3. 显示"正在检查更新..."
4. 检查完成：
   - 如果有更新：显示更新对话框
   - 如果没有更新：显示"当前已是最新版本"

### 场景 3: 延迟更新

**用户**: 正在使用应用的用户
**目标**: 不中断当前工作，稍后更新

**流程**:
1. 应用检测到新版本，显示更新通知
2. 用户点击"稍后提醒"
3. 应用继续正常运行
4. 用户关闭应用时，提示"有新版本可用，是否现在安装？"
5. 用户选择"是"，安装更新

### 场景 4: 跳过版本

**用户**: 不想更新到特定版本的用户
**目标**: 跳过某个版本，等待下一个版本

**流程**:
1. 应用检测到新版本 1.1.2
2. 用户查看更新内容后，选择"跳过此版本"
3. 应用记录跳过的版本号
4. 下次检查更新时，如果还是 1.1.2，不再提示
5. 如果检测到 1.1.3，正常提示更新

## 🛠️ 技术需求

### TR1: 技术选型

**更新库**: `electron-updater`
- 官方推荐的 Electron 自动更新解决方案
- 支持 GitHub Releases、S3、自定义服务器等多种更新源
- 内置断点续传、签名验证等功能
- 跨平台支持良好

**更新源**: GitHub Releases
- 项目已使用 GitHub 托管代码
- GitHub Releases 提供稳定的 API
- 支持自动生成更新清单文件

### TR2: 架构设计

**更新服务模块** (`src/main/services/updateService.ts`)
- 封装 `electron-updater` 的功能
- 提供版本检查、下载、安装的统一接口
- 处理更新事件和错误

**IPC 通信**
- 主进程：更新服务运行在主进程
- 渲染进程：通过 IPC 获取更新状态和触发操作
- 事件：`update-available`, `update-downloaded`, `update-error` 等

**设置存储**
- 更新相关设置存储在 `settings` 表中
- 包括：自动检查、检查频率、自动下载、自动安装等

### TR3: GitHub Releases 配置

**Release 文件要求**
- 必须包含对应平台的安装包
- macOS: `.dmg` 或 `.zip` 文件（包含 .app）
- Windows: `.exe` 安装程序（NSIS）
- Linux: `.AppImage`, `.deb`, `.rpm` 文件

**版本标签格式**
- 遵循语义化版本：`v1.1.1`, `v1.1.2` 等
- 与 `package.json` 中的版本号一致

**Release Notes**
- 使用 Markdown 格式
- 包含更新内容、修复列表、已知问题等

### TR4: electron-builder 配置

**发布配置** (`electron-builder.yml`)
```yaml
publish:
  provider: github
  owner: zdhsoft
  repo: xmmusic
  releaseType: release
```

**构建配置**
- 生成更新清单文件（`latest-mac.yml`, `latest.yml` 等）
- 包含文件哈希、大小、下载 URL 等信息
- 支持代码签名（macOS、Windows）

## ⚠️ 约束条件

### C1: 平台限制

- macOS: 需要代码签名才能自动更新（开发版本可能受限）
- Windows: 需要代码签名才能自动更新（可选，但推荐）
- Linux: 不同发行版需要不同的包格式

### C2: GitHub API 限制

- 未认证请求：每小时 60 次
- 认证请求：每小时 5000 次
- 需要合理控制检查频率，避免触发限制

### C3: 网络环境

- 需要用户设备能够访问 GitHub
- **中国地区网络限制**（重要）:
  - GitHub 在中国大陆地区访问可能不稳定，存在被墙的风险
  - GitHub Releases API 和下载可能受到网络限制
  - 某些网络环境下可能完全无法访问 GitHub
  - 需要提供备选方案和友好的错误提示
- 某些地区可能需要代理或镜像源（未来扩展）

### C4: 用户权限

- macOS: 安装更新需要管理员权限（首次）
- Windows: 安装更新需要管理员权限
- Linux: 安装更新需要 sudo 权限

## 🌏 中国地区网络限制分析与解决方案

### 问题分析

**现状**:
- GitHub 在中国大陆地区访问不稳定，存在被墙风险
- GitHub Releases API 和下载可能受到网络限制
- 某些网络环境下可能完全无法访问 GitHub
- 用户可能使用代理，也可能没有代理

**影响**:
- 自动更新功能可能无法正常工作
- 用户体验受影响
- 需要提供友好的错误提示和备选方案

### 解决方案

#### 方案 1: 多更新源支持（推荐）

**实现方式**:
1. **主更新源**: GitHub Releases（默认）
2. **备用更新源**:
   - Gitee Releases（国内镜像）
   - 自建 CDN（可选）
   - 自定义服务器（可选）

**重要说明 - Gitee CI/CD 限制**:
- ⚠️ **Gitee 目前没有类似 GitHub Actions 的内置 CI/CD 功能**
- Gitee 不支持自动构建和发布 Release
- 需要手动或通过其他方式同步 Release 到 Gitee

**Gitee Release 同步方案**:
1. **方案 A: 使用 GitHub Actions 同步到 Gitee**（推荐）
   - 在 GitHub Actions 工作流中，构建完成后自动同步 Release 到 Gitee
   - 使用 `git-mirror-action` 或自定义脚本
   - 优点：自动化，无需手动操作
   - 缺点：依赖 GitHub Actions

2. **方案 B: 手动同步 Release**
   - GitHub 发布 Release 后，手动在 Gitee 创建 Release
   - 上传相同的安装包文件
   - 优点：简单直接
   - 缺点：需要手动操作，容易遗漏

3. **方案 C: 使用 Webhook + 自建服务**
   - GitHub Release 创建时触发 Webhook
   - 自建服务自动同步到 Gitee
   - 优点：可控性强
   - 缺点：需要维护服务器

**技术实现**:
```typescript
// 更新源优先级
const updateSources = [
  { provider: 'github', owner: 'zdhsoft', repo: 'xmmusic' },
  { provider: 'generic', url: 'https://gitee.com/zdhsoft/xmmusic/releases' },
  // 可以添加更多备用源
]

// 按顺序尝试，失败后切换到下一个
```

**优点**:
- 提高更新成功率
- 用户体验更好
- 不依赖单一服务

**缺点**:
- 需要维护多个更新源
- Gitee Release 需要手动或通过 GitHub Actions 同步
- 实现复杂度较高

---

#### 方案 2: 代理检测和提示（中期方案）

**实现方式**:
1. 检测网络连接状态
2. 如果 GitHub 无法访问，检测是否配置了代理
3. 提供代理配置指南
4. 提供手动更新选项

**技术实现**:
```typescript
// 检测 GitHub 可访问性
async function checkGitHubAccessibility(): Promise<boolean> {
  try {
    const response = await fetch('https://api.github.com', {
      timeout: 5000
    })
    return response.ok
  } catch {
    return false
  }
}

// 如果无法访问，提示用户
if (!await checkGitHubAccessibility()) {
  showProxyGuideDialog()
}
```

**优点**:
- 实现简单
- 引导用户解决问题

**缺点**:
- 依赖用户配置代理
- 不是所有用户都有代理

---

#### 方案 3: 手动更新增强（基础方案）

**实现方式**:
1. 检测到网络问题时，提供清晰的手动更新指引
2. 在设置页面提供"检查更新"按钮，失败时显示手动下载链接
3. 提供国内镜像下载链接（如 Gitee Releases）

**技术实现**:
```typescript
// 更新检查失败时
if (error.code === 'ENOTFOUND' || error.code === 'ETIMEDOUT') {
  showManualUpdateDialog({
    githubUrl: 'https://github.com/rosesmall2010/xmmusic/releases',
    giteeUrl: 'https://gitee.com/zdhsoft/xmmusic/releases', // 备用
    currentVersion: app.getVersion()
  })
}
```

**优点**:
- 实现最简单
- 不依赖网络环境
- 用户可以选择下载方式

**缺点**:
- 需要用户手动操作
- 体验不如自动更新

---

#### 方案 4: 使用国内 CDN（长期方案）

**实现方式**:
1. 将更新包同步到国内 CDN（如阿里云 OSS、腾讯云 COS）
2. 配置多个 CDN 节点
3. 自动选择最快的 CDN

**优点**:
- 访问速度快
- 稳定性高
- 用户体验好

**缺点**:
- 需要额外成本
- 需要维护 CDN 配置

### 错误处理策略

**网络错误处理**:
```typescript
// 更新检查失败时的处理
try {
  await autoUpdater.checkForUpdates()
} catch (error) {
  if (isNetworkError(error)) {
    // 网络错误：提供手动更新选项
    showNetworkErrorDialog({
      message: '无法连接到更新服务器',
      manualUpdateUrl: 'https://github.com/rosesmall2010/xmmusic/releases',
      giteeUrl: 'https://gitee.com/zdhsoft/xmmusic/releases'
    })
  } else {
    // 其他错误：记录日志，不影响应用使用
    console.error('更新检查失败:', error)
  }
}
```

**用户提示文案**:
- 中文: "无法连接到更新服务器，可能是网络问题。您可以：\n1. 检查网络连接\n2. 配置代理后重试\n3. 手动下载更新：{链接}"
- 英文: "Unable to connect to update server. You can:\n1. Check your network connection\n2. Configure proxy and retry\n3. Download manually: {link}"
