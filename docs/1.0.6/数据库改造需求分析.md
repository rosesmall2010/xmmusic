# 数据库改造需求分析报告

**分析日期**: 2025-12-10
**分析人**: Mary (Business Analyst)
**版本**: v1.0.6

## 执行摘要

本次数据库改造是一个**重大架构重构**，涉及核心数据模型的根本性改变。主要目标是通过引入目录表（`music_dir`）来优化文件路径存储，并统一所有音乐列表基于 `all_music` 表的 ID 关联。这是一个**破坏性变更**，需要放弃现有数据库，重新构建。

---

## 一、需求概述

### 1.1 核心变更点

| 变更项 | 当前状态 | 目标状态 | 影响级别 |
|--------|---------|---------|---------|
| 数据库名称 | `xm.db` / `xm-dev.db` | `m.db` / `m-dev.db` | 🔴 高 |
| 文件路径存储 | 完整路径字符串 | 目录ID + 文件名 | 🔴 高 |
| 音乐表名称 | `music` | `all_music` | 🟡 中 |
| 列表关联方式 | 基于 `file_path` 独立存储 | 基于 `all_music.id` 关联 | 🔴 高 |
| 目录管理 | `music_directory` (无限制) | `local_music_dir` (最多20个) | 🟡 中 |

### 1.2 关键设计决策

1. **放弃数据库升级**：不进行数据迁移，直接重建数据库
2. **路径优化**：通过目录表缩短 `file_path` 存储长度
3. **统一关联**：所有列表统一使用 `all_music.id` 作为关联键
4. **文件状态追踪**：新增 `is_exists` 和播放状态字段

### 2.2 新增 `music_dir` 表（音乐目录表）

**需求描述**：
- 表名：`music_dir`
- 目的：存储 `all_music` 表中所有音乐文件所在的目录路径，缩短 `all_music` 表中 `file_path` 的长度
- **关键理解**：`music_dir` 存储的是**所有实际包含音乐文件的目录**（包括子目录），因为同一个目录下会有大量音乐文件，通过目录ID来优化存储
- 字段：
  - `id`：自增主键（INTEGER PRIMARY KEY AUTOINCREMENT）
  - `path`：完整目录路径（TEXT UNIQUE NOT NULL）

**设计分析**：

#### ✅ 优点
1. **存储优化**：避免重复存储相同目录路径，同一目录下的多个文件共享一个 `dir_id`
2. **查询性能**：通过目录ID关联，减少字符串比较
3. **路径管理**：集中管理目录信息，便于维护
4. **空间节省**：对于包含大量文件的目录，显著减少存储空间

#### ⚠️ 潜在问题
1. **路径分隔符**：Windows (`\`) vs macOS/Linux (`/`) 需要统一处理
2. **路径规范化**：需要确保路径格式一致（末尾是否带分隔符）
3. **目录变更**：如果目录被移动/重命名，需要更新 `music_dir` 表
4. **子目录处理**：扫描时会递归扫描子目录，所有子目录都会存入 `music_dir` 表

#### 📝 扫描流程说明
```
用户配置扫描目录（local_music_dir）
    ↓
例如：dir1, dir2
    ↓
递归扫描 dir1 和 dir2 下的所有子目录
    ↓
扫描到的所有目录（dir1, dir1/sub1, dir1/sub2, dir2, dir2/sub1...）
    ↓
全部存入 music_dir 表
    ↓
扫描到的音乐文件存入 all_music 表，通过 dir_id 关联到对应的 music_dir
```

**建议表结构**：
```sql
CREATE TABLE IF NOT EXISTS music_dir (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  path TEXT UNIQUE NOT NULL,
  -- 路径规范化：统一使用正斜杠，去除末尾分隔符
  -- Windows: C:\Music → C:/Music
  -- macOS: /Users/name/Music → /Users/name/Music
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_music_dir_path ON music_dir(path);
```

**关键实现点**：
- 路径规范化函数：统一处理不同操作系统的路径格式
- 路径比较：使用规范化后的路径进行比较
- 目录ID获取：插入或查询时获取目录ID

### 2.3 重构 `all_music` 表（原 `music` 表）

**需求描述**：
- 表名：`music` → `all_music`
- 文件路径存储方式：`dir_id` + `file_name`（而非完整路径）
- 新增字段：
  - `dir_id`：目录ID（外键关联 `music_dir.id`）
  - `file_name`：文件名（不含路径）
  - `is_exists`：文件是否存在（INTEGER，1=存在，0=不存在）
  - `is_playable`：是否可以播放（INTEGER，1=可播放，0=不可播放）
  - `play_error_reason`：不能播放的原因（TEXT）

**当前表结构**（`music`）：
```sql
CREATE TABLE music (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  title TEXT NOT NULL,
  artist TEXT NOT NULL,
  album TEXT,
  -- ... 其他字段
  file_path TEXT UNIQUE NOT NULL,  -- 完整路径
  file_name TEXT NOT NULL,
  -- ...
);
```

**目标表结构**（`all_music`）：
```sql
CREATE TABLE IF NOT EXISTS all_music (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  dir_id INTEGER NOT NULL,
  file_name TEXT NOT NULL,
  -- 完整路径 = music_dir.path + 分隔符 + file_name
  -- Windows: music_dir.path + '\' + file_name
  -- macOS/Linux: music_dir.path + '/' + file_name

  title TEXT NOT NULL,
  artist TEXT NOT NULL,
  album TEXT,
  year INTEGER,
  genre TEXT,
  file_size INTEGER NOT NULL,
  file_hash TEXT NOT NULL,
  file_extension TEXT NOT NULL,
  duration INTEGER,
  bitrate INTEGER,
  sample_rate INTEGER,
  channels INTEGER,
  cover_path TEXT,  -- 也使用 dir_id + 文件名格式？
  lyrics_path TEXT,  -- 也使用 dir_id + 文件名格式？

  -- 新增字段
  is_exists INTEGER DEFAULT 1,  -- 文件是否存在
  is_playable INTEGER DEFAULT 1,  -- 是否可以播放
  play_error_reason TEXT,  -- 不能播放的原因

  play_count INTEGER DEFAULT 0,
  last_played_at DATETIME,
  added_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  is_corrupted INTEGER DEFAULT 0,
  is_duplicate INTEGER DEFAULT 0,

  FOREIGN KEY (dir_id) REFERENCES music_dir(id) ON DELETE RESTRICT,
  UNIQUE(dir_id, file_name)  -- 唯一性约束：同一目录下文件名唯一
);
```

**关键设计决策**：

#### 1. 文件路径唯一性判断
- **当前**：通过 `file_path` (完整路径) 判断
- **目标**：通过 `(dir_id, file_name)` 组合判断
- **实现**：使用 `UNIQUE(dir_id, file_name)` 约束

#### 2. 完整路径构建
需要提供工具函数：
```typescript
function buildFullPath(dirPath: string, fileName: string, platform: NodeJS.Platform): string {
  const separator = platform === 'win32' ? '\\' : '/'
  // 确保 dirPath 不以分隔符结尾
  const normalizedDir = dirPath.replace(/[/\\]+$/, '')
  return `${normalizedDir}${separator}${fileName}`
}
```

#### 3. 封面和歌词路径
需求中提到："封面和歌词的文件路径也用目录id+文件组成"
- **问题**：封面和歌词可能不在音乐文件同一目录
- **建议**：
  - 方案A：如果封面/歌词与音乐文件同目录，使用 `dir_id` + 文件名
  - 方案B：如果封面/歌词在不同目录，需要单独的目录ID或完整路径
  - **需要澄清**：封面和歌词的存储位置策略

#### 4. `is_exists` 字段维护
- **何时更新**：文件扫描时、播放前检查
- **更新策略**：定期扫描或按需检查
- **性能考虑**：大量文件存在性检查可能耗时

#### 5. `is_playable` 和 `play_error_reason` 字段
- **何时设置**：首次扫描、播放失败时
- **错误原因示例**：
  - "文件损坏"
  - "格式不支持"
  - "编码错误"
  - "文件不存在"

### 2.4 新增 `local_music_dir` 表（自动扫描目录配置表）

**需求描述**：
- 表名：`local_music_dir`
- 目的：记录用户配置的**自动扫描根目录列表**，最多20个目录
- 功能：需要提供增删改查功能
- 扫描行为：本地音乐扫描时，会**递归扫描**这些目录下的所有文件

**与 `music_dir` 的关系**：

| 表名 | 用途 | 存储内容 | 关系 |
|------|------|---------|------|
| `local_music_dir` | **扫描配置** | 用户配置的扫描根目录（最多20个） | 例如：dir1, dir2 |
| `music_dir` | **目录存储** | `all_music` 表中所有音乐文件所在的目录（包括子目录） | 扫描时自动填充 |

**扫描流程示例**：
```
用户配置 local_music_dir：
  - dir1: /Users/name/Music
  - dir2: /Users/name/Downloads/Music

扫描时：
  1. 递归扫描 dir1 下的所有子目录和文件
     → 发现：/Users/name/Music/Rock/album1.mp3
     → 发现：/Users/name/Music/Jazz/album2.mp3
     → 发现：/Users/name/Music/Pop/subfolder/song.mp3

  2. 递归扫描 dir2 下的所有子目录和文件
     → 发现：/Users/name/Downloads/Music/song1.mp3

  3. 所有发现的目录存入 music_dir：
     - /Users/name/Music/Rock
     - /Users/name/Music/Jazz
     - /Users/name/Music/Pop/subfolder
     - /Users/name/Downloads/Music

  4. 所有发现的文件存入 all_music，通过 dir_id 关联到对应的 music_dir
```

**建议表结构**：
```sql
CREATE TABLE IF NOT EXISTS local_music_dir (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  path TEXT UNIQUE NOT NULL,  -- 扫描根目录路径（完整路径）
  display_order INTEGER DEFAULT 0,  -- 显示顺序
  enabled INTEGER DEFAULT 1,  -- 是否启用扫描
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 限制最多20条记录（应用层控制）
CREATE INDEX IF NOT EXISTS idx_local_music_dir_order ON local_music_dir(display_order);
CREATE INDEX IF NOT EXISTS idx_local_music_dir_enabled ON local_music_dir(enabled);
```

**⚠️ 重要设计决策**：
- `local_music_dir` **不关联** `music_dir.id`，而是直接存储路径
- 原因：`local_music_dir` 存储的是**扫描根目录**，而 `music_dir` 存储的是**所有子目录**
- 扫描时：读取 `local_music_dir` 的路径 → 递归扫描 → 发现的目录存入 `music_dir` → 发现的文件存入 `all_music`

**扫描逻辑伪代码**：
```typescript
async function scanMusicDirectories() {
  // 1. 获取所有启用的扫描根目录
  const scanRoots = await db.getLocalMusicDirs({ enabled: true })

  for (const root of scanRoots) {
    // 2. 递归扫描目录下的所有文件和子目录
    const files = await recursiveScan(root.path)

    for (const file of files) {
      const dirPath = path.dirname(file.path)
      const fileName = path.basename(file.path)

      // 3. 获取或创建目录记录（存入 music_dir）
      const dirId = await db.getOrCreateMusicDir(dirPath)

      // 4. 插入或更新音乐文件（存入 all_music）
      await db.upsertMusic({
        dir_id: dirId,
        file_name: fileName,
        // ... 其他元数据
      })
    }
  }
}
```

**功能需求**：
1. **添加扫描目录**：
   - 检查路径是否已存在
   - 检查是否超过20个限制
   - 验证路径是否存在且可访问
2. **删除扫描目录**：
   - 删除 `local_music_dir` 记录
   - 可选：清理 `music_dir` 和 `all_music` 中相关的数据
3. **修改扫描目录**：
   - 更新显示顺序
   - 启用/禁用扫描
4. **查询扫描目录**：
   - 按显示顺序排序
   - 只返回启用的目录

**与现有 `music_directory` 表的区别**：

| 特性 | `music_directory` (当前) | `local_music_dir` (目标) |
|------|-------------------------|-------------------------|
| 记录数限制 | 无限制 | 最多20个 |
| 存储内容 | 完整目录信息（路径、扫描配置等） | 仅存储扫描根目录路径 |
| 用途 | 扫描配置（可能包含子目录） | 扫描根目录列表 |
| 扫描行为 | 根据配置扫描 | 递归扫描所有子目录 |
| 关联关系 | 独立表 | 不关联 `music_dir`，直接存储路径 |

### 2.5 列表关联方式变更

**当前架构**（v1.0.5 列表独立性）：
- 所有列表（`local_music`, `favorites`, `playlist_item`, `recent_plays` 等）独立存储 `file_path`
- 通过 `file_path` 或 `file_path_md5` 关联 `music` 表
- **优点**：即使 `music` 表重建，列表数据不丢失
- **缺点**：关联查询需要字符串匹配，性能较低

**目标架构**（v1.0.6）：
- 所有列表基于 `all_music.id` 关联
- 列表表只存储 `music_id`（外键）
- **优点**：关联查询性能高，数据一致性好
- **缺点**：如果 `all_music` 表重建，需要重新关联（但需求中已明确放弃升级）

**需要修改的表**：

#### 1. `local_music` 表
```sql
-- 当前结构
CREATE TABLE local_music (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  file_path TEXT NOT NULL UNIQUE,
  file_path_md5 TEXT NOT NULL,
  added_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 目标结构
CREATE TABLE IF NOT EXISTS local_music (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  music_id INTEGER NOT NULL UNIQUE,  -- 关联 all_music.id
  added_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (music_id) REFERENCES all_music(id) ON DELETE CASCADE
);
```

#### 2. `favorites` 表
```sql
-- 当前结构
CREATE TABLE favorites (
  id INTEGER PRIMARY KEY AUTOINCREMENT,

  added_at DATETIME
);

-- 目标结构
CREATE TABLE IF NOT EXISTS favorites (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  music_id INTEGER NOT NULL UNIQUE,  -- 关联 all_music.id
  added_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (music_id) REFERENCES all_music(id) ON DELETE CASCADE
```

#### 3. `playlist_item` 表
```sql
-- 当前结构
CREATE TABLE playlist_item (

CREATE TABLE IF NOT EXISTS playlist_item (
```

```typescript
async function updateMusicMetadata(musicId: number, metadata: {
```

## 三、架构对比分析

### 3.1 当前架构（v1.0.5）

**特点**：
- ✅ 列表独立性：列表数据不依赖 `music` 表
- ✅ 数据恢复：即使 `music` 表重建，列表数据保留
- ⚠️ 关联查询：需要字符串匹配，性能较低
- ⚠️ 数据一致性：可能出现列表中的文件在 `music` 表中不存在的情况

**数据流**：
```
文件扫描 → music 表（存储元数据）
                ↓
        列表表（独立存储 file_path）
                ↓
        通过 file_path 关联查询
```

### 3.2 目标架构（v1.0.6）

**特点**：
- ✅ 关联性能：通过整数ID关联，查询速度快
- ✅ 数据一致性：外键约束保证数据完整性
- ✅ 存储优化：目录表减少重复路径存储
- ⚠️ 依赖性强：列表完全依赖 `all_music` 表
- ⚠️ 数据恢复：如果 `all_music` 表重建，需要重新扫描

**数据流**：
```
用户配置扫描目录
    ↓
local_music_dir 表（存储扫描根目录，最多20个）
    ↓
递归扫描这些目录下的所有文件和子目录
    ↓
发现的目录 → music_dir 表（存储所有包含音乐文件的目录）
发现的文件 → all_music 表（dir_id + file_name，关联到 music_dir）
    ↓
列表表（local_music, favorites, playlist_item 等）
    ↓
存储 music_id（关联到 all_music.id）
    ↓
通过 music_id 关联查询，性能高
```

**关键关系说明**：
- `local_music_dir`：用户配置的扫描根目录（例如：`/Users/name/Music`）
- `music_dir`：扫描过程中发现的所有目录（包括子目录，例如：`/Users/name/Music/Rock`, `/Users/name/Music/Jazz`）
- `all_music`：所有音乐文件，通过 `dir_id` 关联到 `music_dir`
- 列表表：通过 `music_id` 关联到 `all_music`
