# 数据库改造需求列表
- 现在的数据库放弃，不做数据库升级。新调整数据库名： m.db, m-dev.db
- 增加一个音乐目录表：music_dir，记录所有音乐目录。音乐目录表有id，path, path是一个完整目录路径。它的目的是缩短后面all_music表的file_path的长度， 每个文件分为dir + file_name.(其中这个id是自增的)
- 有一个music总表: all_music，包含音乐的音乐用到的字段，其中id是主键，自增的整数。
  - filepath包含2部分，一个是目录id(music_dir的id), 一个是文件名：music_dir的path+分隔符+文件名是音乐文件的完整全路径，不是相对路径：例如： windows 系统下：c:\music\1.mp3，mac 系统下：/music/1.mp3。一般情况下：windows的分隔符是\，macos和linux的分隔符是/
  - 判断音乐文件是否是同一个文件，通过文件的filepath(目录id+文件名)来判断。
  - 这个表增加一个字段 is_exists，表示改音乐文件是否存在。如果文件存在，在标记为1，否则标记为0。
  - 增加2个字段，表示这个文件是否可以正常播放，如果可以正常播放，在标记为1，否则标记为0， 以及不能播放的原因。
  - 这个表有音乐的基本信息:歌手，专辑，封面， 码率，时长，歌词，tag信息等其他必要信息
  - 其他各个音乐表，都是基于all_music表，通过id来关联的。例如我的歌单，收藏，我的歌单等。
  - 封面和歌词的文件路径也用目录id+文件组成。
- 本地音乐目录表：local_music_dir，记录当前音乐目录的，最多20个目录。需要增加本地音乐目录增删改查功能。本地音乐扫描的时候，要求扫描这些目录。
- app逻辑的文件信息，全部基于all_music和music_dir：app的发现音乐、本地音乐、我喜欢、最近播放、我的歌单（各个自定义歌单），当前播放列表都是独立的音乐列表，都是基于all_music的上的，通过all_music的id关联。
- 修改标签的时候，要同步更新all_music的信息和对应音乐列表和对应的界面。
