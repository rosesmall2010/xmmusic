## 📝 详细需求分析

### REQ-1: 数据库版本控制机制

#### 1.1 需求描述
- 增加数据库版本控制功能
- 当数据库版本与代码要求的版本不一致时，自动清空并重建数据库
- 清空时删除所有数据和文件
- 版本号在代码中定义，由人工手动修改
- 初始化数据库时保存版本号，用于未来对比

#### 1.2 业务规则
- **版本定义位置**: 版本号定义在 `dbver.ts` 文件中（如 `export const DB_VERSION = '1.0.5'`）
- **版本存储**: 数据库 `settings` 表中存储 `db_version` 键值对
- **版本检查时机**: 数据库初始化时检查
- **版本不匹配处理**:
  - 版本号不相同或不存在时，执行重建流程
  - 删除现有数据库文件
  - 删除所有关联数据文件（如封面、歌词等）
  - 重建数据库结构
  - 设置新版本号到 `settings` 表中
- **重要**: 根据版本控制机制，不需要迁移数据，版本不匹配时自动清空并重建数据库

#### 1.3 数据结构设计

```sql
-- settings 表中存储
key: 'db_version'
value: '1.0.5'  -- JSON 字符串或直接字符串
```

#### 1.4 技术实现要点
- 在 `db.ts` 的 `initialize()` 方法中添加版本检查逻辑
- 从 `dbver.ts` 文件导入版本号
- 从数据库 `settings` 表读取已存储的版本号
- 创建版本比较函数
- 创建数据库清理和重建函数
- **版本不匹配时清空重建，不执行数据迁移**

#### 1.5 边界条件
- 首次安装（无版本记录）：视为版本不匹配，执行初始化
- 版本相同：正常启动
- 版本不同：执行重建流程
- 数据库损坏：执行重建流程

### REQ-2: 列表独立性架构

#### 2.1 需求描述
所有列表（本地音乐、发现音乐、我喜欢、最近播放、我的歌单、播放队列）：
- 完全独立，互不依赖
- 以完整路径的 MD5 作为唯一键（key）
- **MD5 计算方式**: 对文件完整路径（`file_path`）进行 MD5 计算，实时计算，不缓存
- MD5 相同的视为同一首歌曲
- 某个列表清空或删除，不影响其他列表

#### 2.2 业务规则
- **唯一标识**: `file_path` → MD5(`file_path`) → 作为列表项的键值
- **MD5 计算**: 实时计算，不存储不缓存，每次需要时从 `file_path` 计算
- **列表存储**: 每个列表独立存储，不依赖 `music` 表
- **删除操作**: 删除列表中的歌曲不影响其他列表

#### 2.3 数据结构设计

**推荐方案: 独立表结构**

```sql
-- 本地音乐列表
CREATE TABLE local_music (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  file_path TEXT NOT NULL UNIQUE,
  added_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 发现音乐列表（DiscoverView，展示最近添加的歌曲）
CREATE TABLE discover_music (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  file_path TEXT NOT NULL,
  discovered_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 我喜欢列表（已存在，需要检查是否符合要求）
CREATE TABLE favorites (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  file_path TEXT NOT NULL UNIQUE,
  added_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 最近播放列表（可以重用 play_history 或创建新表）
CREATE TABLE recent_plays (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  file_path TEXT NOT NULL,
  played_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 播放列表项（已存在，使用 file_path）
-- 保持不变

-- 播放队列（新表，持久化存储）
CREATE TABLE play_queue (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  file_path TEXT NOT NULL,
  position INTEGER NOT NULL,
  added_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**注意**:
- 不需要在表中存储 `file_path_md5` 字段
- MD5 值实时计算，用于查询和比较
- 列表通过 `file_path` 关联到 `music` 表获取元数据

#### 2.4 MD5 计算实现

```typescript
// 工具函数：计算文件路径的 MD5
function calculateFilePathMD5(filePath: string): string {
  const crypto = require('crypto')
  return crypto.createHash('md5').update(filePath).digest('hex')
}

// 使用场景：
// 1. 列表项比较（判断是否为同一首歌曲）
// 2. 列表项去重
// 3. 查询时作为条件
```

#### 2.5 数据初始化策略
- **不需要数据迁移**：根据版本控制机制，版本不匹配时自动清空并重建数据库
- 新数据库初始化时创建所有新的表结构
- 不需要预先计算 MD5，使用时实时计算
- 用户重新扫描和添加音乐到各个列表

#### 2.6 API 接口变更
所有列表操作接口需要：
- 支持基于 `file_path` 的操作
- 使用 MD5 进行列表项比较和去重（实时计算）
- 列表间完全隔离

### REQ-3: 列表显示与数据加载

#### 3.1 需求描述
所有列表显示时：
- 从数据库直接读取
- 显示字段：歌曲名称、专辑、完整路径、乐曲ID（文件路径的MD5，实时计算）、封面信息、自增ID、加入时间
- 列表顺序：按自增ID从小到大（即加入时间顺序）
- 分页加载：总数量从数据库读取，滚动条和翻页按总数量调整
- 按需加载：缓存机制，当前页未缓存时从数据库读取
- **清空列表后**: 自增ID不重置，继续累加

#### 3.2 业务规则
- **显示字段列表**:
  - `title` (歌曲名称) - 从 `music` 表 JOIN 获取
  - `album` (专辑) - 从 `music` 表 JOIN 获取
  - `file_path` (完整路径) - 从列表表获取
  - `file_path_md5` (乐曲ID) - 实时计算 MD5(file_path)
  - `cover_path` (封面信息) - 从 `music` 表 JOIN 获取
  - `id` (自增ID) - 从列表表获取
  - `added_at` (加入时间) - 从列表表获取

- **排序规则**: `ORDER BY id ASC`（自增ID从小到大）

- **分页策略**:
  - 每页显示数量：20 条（可配置）
  - 预加载策略：读取当前页 + 前后各1页（即读取20+1+1=22条）
  - 缓存管理：前端维护分页缓存
  - 滚动触发：接近页面底部时预加载下一页

- **自增ID规则**:
  - 清空列表后不自增ID，继续累加
  - 例如：列表有 100 条记录（ID: 1-100），清空后，新添加的歌曲 ID 从 101 开始

#### 3.3 数据结构设计

```typescript
interface ListItem {
  id: number                    // 列表自增ID（不清空）
  filePath: string             // 完整路径
  filePathMd5: string          // 路径MD5（实时计算，不存储）
  title: string                // 歌曲名称（JOIN music表）
  album: string | null         // 专辑（JOIN music表）
  coverPath: string | null     // 封面路径（JOIN music表）
  addedAt: string              // 加入时间
  // 其他元数据字段（可选，按需JOIN）
}
```

#### 3.4 查询优化
- 使用 LEFT JOIN 一次性获取列表数据和音乐元数据
- 为 `file_path` 创建索引
- 使用 LIMIT/OFFSET 实现分页
- 考虑使用虚拟滚动优化大列表性能

### REQ-4: 标签编辑全局同步

#### 4.1 需求描述
在某个列表（如歌单）中右键"编辑标签"并保存后：
- 所有包含该歌曲的列表立即更新标签显示
- 如果该歌曲正在显示（如当前列表、正在播放UI、播放队列），立即更新显示
- 数据库中的所有相关记录同步更新

#### 4.2 业务规则
- **标签存储位置**: `music` 表（作为音乐元数据的主表）
- **同步范围**:
  - 所有包含该 `file_path` 的列表
  - 当前正在播放的UI
  - 播放队列显示（两个位置：Footer PlayQueueDrawer、NowPlayingView 右侧面板）
- **更新时机**: 标签保存后立即触发

#### 4.3 数据结构设计
标签信息存储在 `music` 表中：
- `title`, `artist`, `album`, `year`, `genre`, `cover_path` 等

#### 4.4 同步机制
1. **数据库更新**: 更新 `music` 表中对应 `file_path` 的记录
2. **事件通知**: 通过 IPC 或事件系统通知前端
3. **前端更新**:
   - 更新所有显示该歌曲的组件
   - 更新当前播放UI
   - 更新播放队列显示（两个位置）
   - 更新缓存中的数据

#### 4.5 实现要点
- 标签编辑API需要返回更新后的完整音乐信息
- 前端需要监听标签更新事件
- 所有列表组件需要支持动态更新单个项目
- 缓存系统需要支持部分更新

### REQ-6: 列表项快捷操作功能（"我喜欢"和"播放队列"图标）

#### 6.1 需求描述

##### 6.1.1 "我喜欢"功能
除了"我喜欢"列表本身，其他所有列表都支持添加到"我喜欢"：
- 列表项增加"我喜欢"图标按钮
- 未喜欢时：灰色图标
- 已喜欢时：亮色图标
- 添加到"我喜欢"时：
  - 同步更新"我喜欢"列表
  - 更新"我喜欢"菜单的总数显示
- 本地音乐批量操作：增加"添加到我喜欢"选项

##### 6.1.2 "播放队列"功能
除了播放队列本身，其他所有列表都支持添加到播放队列：
- 列表项增加"播放队列"图标按钮，在"我喜欢"按钮的后面
- 不在队列：灰色图标
- 在队列：亮色图标
- **点击行为**:
  - 灰色时：点击后将当前歌曲添加到播放队列（加到队列尾部）
  - 亮色时：点击后将当前歌曲从播放队列中移除（从队列中移除，如果是当前歌曲，切换到下一首歌曲）
- **双击行为**：播放该歌曲
  - 如果不在队列：则加到队列尾部并播放
  - 如果已经在队列：则切换到该歌曲并播放

##### 6.2.1 "我喜欢"功能规则
- **支持添加到"我喜欢"的列表**:
  - ✅ 本地音乐
  - ✅ 我的歌单（各歌单）
  - ✅ 播放队列（Footer PlayQueueDrawer、NowPlayingView 右侧面板）
  - ✅ 当前正在播放
  - ✅ 发现音乐（DiscoverView）
  - ✅ 最近播放
  - ❌ 我喜欢（本身不需要）

- **"我喜欢"操作反馈**:
  - 点击后立即更新图标状态
  - 同步更新"我喜欢"列表（如果当前在查看）
  - 更新菜单栏的计数显示

##### 6.2.2 "播放队列"功能规则
- **支持添加到播放队列的列表**:
  - ✅ 本地音乐
  - ✅ 我的歌单（各歌单）

  - ✅ 当前正在播放
  - ✅ 发现音乐（DiscoverView）

  - ❌ 播放队列（本身不需要）

  - 点击后立即更新图标状态
  - 同步更新播放队列显示（两个位置：Footer PlayQueueDrawer、NowPlayingView 右侧面板）

- **队列状态判断**:
  - 通过实时计算 `file_path` 的 MD5 判断歌曲是否在队列中

## 🔍 影响分析

### 数据库架构影响

#### 现有表结构影响
- ✅ `music` 表：保持不变，作为音乐元数据主表
- ✅ `playlist_item` 表：已使用 `file_path`，符合要求
- ✅ `favorites` 表：已存在，符合要求
- ⚠️ `play_history` 表：需要重构或创建新的 `recent_plays` 表
- ❌ 需要新建：`local_music`、`discover_music`、`play_queue` 表

#### 数据迁移影响
- **不需要迁移数据**：之前的数据根据版本控制机制，会自动清空并重建数据库
- **版本控制机制**：
  - 版本号定义在 `dbver.ts` 文件中（如 `export const DB_VERSION = '1.0.5'`）
  - 数据库 `settings` 表中也存放这个版本号
  - 版本号不相同或不存在时，重建数据库
- **数据库重建流程**：
  - 删除现有数据库文件
  - 删除所有关联数据文件（封面、歌词等）
  - 重建数据库结构（所有新表）
  - 设置新版本号到 `settings` 表
- **用户数据**：用户需要重新扫描音乐文件并添加到各个列表
- **不需要预计算 MD5**：使用时实时计算

### 性能影响

#### 优化点
- 分页加载减少内存占用
- 缓存机制提升加载速度
- 异步扫描提升响应速度
- 批量操作提升操作效率

#### 潜在风险
- MD5 实时计算可能影响性能（需要优化计算函数）

- 在必要时可以考虑前端缓存计算结果（仅用于显示，不作为数据库键）
- 优化 JOIN 查询，使用合适的索引
